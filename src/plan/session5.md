# [교안] 5차시. 조건 분기 & 판단형 Agent 설계

## 1. 개요 및 목표

- **대상**: 사용자의 입력이나 상황 판단에 따라 서로 다른 후속 작업을 수행하는 동적 에이전트를 만들고자 하는 학습자
- **시간**: 총 60분 (도입 5분, 설명 20분, 실습 25분, 공유 및 정리 10분)
- **목표**:
  - 에이전트 설계에서 조건 분기(if-else)의 중요성과 활용 목적을 이해한다.
  - 분류 기준(긍정, 부정, 문의, 복합)과 명확한 판단 근거를 프롬프트에 정의한다.
  - 애매한 정보나 복합 의도를 처리하기 위한 예외 처리 규칙을 수립하고 반영한다.

## 3. 세부 강의 가이드

### [00:00 - 00:05] 도입: 생각 열기 (5분)

#### 강사 설명 가이드
> "자동 응답 챗봇을 설계했는데, 고객이 '정말 친절하고 배송도 빨라요!'라고 한 칭찬 리뷰와 '환불해 주세요. 너무 화가 나네요'라는 불만 리뷰에 똑같이 '감사합니다, 담당자에게 전달하겠습니다'라고 답변하면 어떻게 될까요? 불만을 제기한 고객은 더욱 화가 나겠죠. 이처럼 실무 에이전트는 상황을 **'판단(Reasoning)'**하고 그에 따라 처리 방향을 **'분기(Branching)'**해야 합니다."

#### 도입 질문
- *"고객의 이메일을 자동으로 읽고 처리하는 에이전트가 있다면, 메일의 목적에 따라 어떤 다른 흐름으로 나눠야 할까요?"*
- *"만약 AI가 분류하기에 애매한 질문(예: '재입고 언제 하나요? 근데 이번 신제품 예쁘네요'와 같이 칭찬과 질문이 섞인 글)이 들어온다면 어떻게 처리하는 것이 현명할까요?"*

### [00:05 - 00:25] 핵심 설명: 조건 분기와 예외 처리 설계 (20분)

#### 1) 조건 분기(Conditional Branching)의 개념
- **정의**: 입력값이나 상황 분석 결과에 따라 서로 다른 업무 파이프라인(Flow)으로 흘러가게 만드는 구조입니다.
- **분류 기준의 설계**: 기준이 모호하면 에이전트의 판단 정확도가 급감합니다.
  - **긍정**: 만족, 칭찬, 추천, 재구매 의사 표현
  - **부정**: 불만, 하자에 대한 항의, 오류 접수, 환불/교환 요구
  - **문의**: 일정, 방법, 가격, 절차 등에 대한 정보 요청
  - **복합/예외**: 긍정과 부정이 섞여 있거나, 글이 짧아 의도를 알 수 없는 경우

#### 시각 자료 1. 조건 분기 의사결정 흐름도 (Decision Tree)
```
                  [ 고객 입력 (리뷰/메일) ]
                              │
                              ▼
                    { 의도 분류 판단 (LLM) }
                   /          │           \
            [ 긍정 ]       [ 부정 ]       [ 문의 ]
               │              │              │
               ▼              ▼              ▼
         [감사 답변]    [긴급 담당자]   [FAQ 답변]
                        [배정 및 알림]   [자동 발송]
```
*그림 설명: 입력값 하나가 들어왔을 때 LLM이 3개의 갈래로 판단을 내리고 각각 다른 액션 카드로 향하는 트리 형태의 다이어그램입니다.*

#### 2) 예외 처리(Exception Handling) 대응법
실무에서는 예상하기 힘든 모호한 입력이 항상 들어옵니다. 이를 위해 프롬프트 안에 명확한 '우선순위 규칙'을 삽입해야 합니다.
- **우선순위 규칙 예시**:
  - 규칙 1: 긍정과 부정이 동시에 들어온 경우(복합), **'부정'**으로 판단하고 불만 요소를 우선 파악한다.
  - 규칙 2: 정보가 부족하거나 외국어가 섞여 해석이 어려우면, 임의로 추측하지 말고 **'추가 확인 필요'**로 분류한다.

#### 시각 자료 2. 예외 처리 가드레일 매핑
```
+───────────────────────────────────────────────────────────+
| [모호한 입력]: "좋아요. 근데 배송이 늦네요." (긍정+부정 복합) |
+───────────────────────────────────────────────────────────+
                             │
                             ▼ [가드레일 규칙]
                   "복합형은 부정을 우선한다"
                             │
                             ▼
                    [최종 분류]: 부정
                    [후속 조치]: 배송팀 즉시 연동
```
*그림 설명: 규칙 카드가 가드레일 역할을 하여 애매한 데이터가 꼬이지 않고 정해진 궤도로 통제되는 과정을 보여줍니다.*

### [00:25 - 00:50] 실습 진행: 리뷰 분류 Agent 설계 (25분)

#### 제공할 실습 텍스트 (리뷰 샘플 5개)
```text
1. "강의 내용이 실무에 바로 적용할 수 있어서 좋았습니다."
2. "신청 페이지 오류 때문에 결제가 두 번 시도됐습니다. 확인 부탁드립니다."
3. "다음 교육 일정은 언제 열리나요?"
4. "강사님 설명은 좋았지만 실습 시간이 조금 부족했습니다."
5. "자료 다운로드 링크가 열리지 않습니다."
```

#### 실습 수행 단계
1. **분류 프롬프트 작성**: 긍정/부정/문의/복합 분류 기준을 RICO 형태로 작성합니다.
2. **이유(Reason) 출력 지정**: 단순 분류명만 출력하지 말고, 왜 그렇게 판단했는지 `판단 이유: [근거 설명]` 항목을 함께 출력하도록 구성합니다.
3. **결과 작성 및 비교**: 5개 리뷰를 분류하고 표 형식으로 요약해 봅니다. 특히 4번과 같이 긍정/부정이 섞인 복합 문장의 결과를 확인하고 예외 규칙을 프롬프트에 보완합니다.

#### 강사용 팁 (Instructor Tip)
> "인공지능에게 판단 결과뿐만 아니라 '판단 근거(이유)'를 함께 적게 하면, AI가 생각(Reasoning)하는 논리가 한 번 정돈되어 분류 정확도 자체가 크게 올라간다는 이론적 사실(CoT와 유사)을 강조해 주세요."
