### Slide 3. 도입 질문: AI가 일을 처리하려면?
- **슬라이드 내용 (Slide Content)**
  - **도입 질문**: "AI가 단순히 답변을 해주는 비서를 넘어, 실제 비즈니스 프로세스 상의 업무를 완료하게 하려면 무엇이 더 필요할까요?"
  - **해결을 위해 필요한 요소**:
    - **상황 판단력 (Reasoning)**: 현재 들어온 데이터가 무엇인지 분류하고 평가하는 힘
    - **도구 사용력 (Action)**: 메일을 발송하거나 데이터베이스에 값을 저장하는 실행력
    - **구조화된 출력 (Output Control)**: 정해진 포맷으로 결과를 전달해 후속 시스템이 이해하도록 하는 안정성
- **발표자 스크립트 (Presenter Script)**
  - "만약 고객의 메일에 답변하는 에이전트를 만든다면, AI는 메일의 톤을 읽어야 하고(판단), 실제로 답변 이메일을 작성해 담당자에게 승인 요청을 보내거나 메일 서버로 발송을 트리거해야 하며(실행), 이 모든 결과를 DB에 남겨야(구조화) 합니다. 즉, 말 잘하는 인공지능이 아니라 일 잘하는 실행력이 필요합니다."
- **시각화 가이드 (Visual Guide)**
  - '판단 -> 도구 사용 -> 출력'의 유기적인 순서도를 가로형 인포그래픽으로 구현.

### Slide 4. 핵심 설명: AI · LLM · Agent의 관계
- **슬라이드 내용 (Slide Content)**
  - **계층 구조 분석**:
    - **AI (인공지능)**: 인간의 지능적 행동을 기계로 구현하려는 가장 넓은 범주의 컴퓨터 과학 분야.
    - **LLM (대형 언어 모델)**: 텍스트 데이터를 대량으로 학습하여 언어의 문맥을 이해하고 자연스럽게 생성하는 핵심 언어 엔진.
    - **Agent (에이전트)**: LLM을 두뇌로 삼아 목표 달성을 위해 판단(Reasoning)하고 행동(Action)을 수행하는 비즈니스 시스템.
- **발표자 스크립트 (Presenter Script)**
  - "이 세 가지 용어가 헷갈리실 수 있습니다. AI는 가장 거대한 숲입니다. 그리고 LLM은 그 숲 안에서 가장 뛰어난 언어적 계산 능력을 가진 최신 나무입니다. 우리가 구축하고자 하는 Agent는 이 LLM이라는 언어적 두뇌를 기반으로, 실제 세상에서 팔을 뻗고 다리를 움직여 목표를 이루는 완성된 로봇과 같은 실행 구조입니다."
- **시각화 가이드 (Visual Guide)**
  - 세 개의 원이 서로를 포함하고 있는 벤다이어그램(AI > LLM > Agent)을 입체적인 레이어로 표현.

### Slide 5. 핵심 설명: 생성형 AI vs 에이전트의 차이점
- **슬라이드 내용 (Slide Content)**
  - **차이 비교표**:
    | 구분 | 생성형 AI (Generative AI) | AI 에이전트 (AI Agent) |
    | :--- | :--- | :--- |
    | **동작 방식** | 사용자 질문에 대해 즉시 답변 생성 | 목표가 주어지면 내부 계획을 수립하여 다단계로 실행 |
    | **사용 주체** | 인간이 직접 개입하여 질문을 보정함 | 스스로 판단하고 도구를 호출하며 인간은 감독함 |
    | **산출물** | 자연어 줄글, 일반적 이미지 등 | 데이터베이스 레코드, 이메일 전송, 완료 보고서 등 |
- **발표자 스크립트 (Presenter Script)**
  - "단순 생성형 AI는 질문을 던지면 바로 답을 주는 단발성 관계입니다. 반면 에이전트는 '이 고객의 환불 요청 건을 최종 처리해줘'라는 목표를 주면, 규칙에 따라 환불 사유를 분류하고, 정책을 확인한 후, 환불 메일을 발송하는 단계를 스스로 기획하여 완료합니다."
- **시각화 가이드 (Visual Guide)**
  - 두 방식의 데이터 흐름 차이를 단순 선형 흐름(생성형 AI)과 순환형 루프(에이전트) 다이어그램으로 대조하여 시각화.

### Slide 6. 핵심 설명: Agent 4대 구성요소 (IRAO)
- **슬라이드 내용 (Slide Content)**
  - **Agent의 기본 작동 구조**:
    - **1. Input (입력)**: 사용자의 명령, 수신된 이메일 텍스트, API를 통해 유입된 로우 데이터.
    - **2. Reasoning (판단/추론)**: 프롬프트에 정의된 Persona와 규칙에 따라 입력값의 의도와 맥락을 분석.
    - **3. Action (행동/도구)**: 코드 실행, 데이터베이스 읽기/쓰기, 이메일 전송 등 외부 API 호출.
    - **4. Output (출력)**: 정제된 텍스트, 사용자용 알림 메시지, 구조화된 JSON 데이터 등 최종 결과물.
- **발표자 스크립트 (Presenter Script)**
  - "에이전트 설계를 지탱하는 4가지 뼈대, 'Input, Reasoning, Action, Output'을 반드시 기억하셔야 합니다. 입력을 받아, 규칙에 의거해 추론 및 판단을 거친 뒤, 필요한 도구를 써서 결과를 외부 시스템이나 사람에게 전달하는 것이 에이전트의 표준 동작 메커니즘입니다."
- **시각화 가이드 (Visual Guide)**
  - IRAO 4개 영역을 각각의 카드 레이아웃으로 구분하고 흐름을 화살표로 연결하여 표현.

## 2차시. 프롬프트 구조화 & Persona 설계

### Slide 13. 핵심 설명: 프롬프트의 4대 뼈대 (R-I-C-O)
- **슬라이드 내용 (Slide Content)**
  - **R-I-C-O 프레임워크**:
    - **Role (역할)**: AI의 신분과 페르소나 정의 (예: 금융 전문 준법감시인)
    - **Instruction (지시사항)**: 수행해야 할 명확하고 구체적인 작업 (예: 약관 검토 및 위반 요소 추출)
    - **Context (배경/맥락)**: 제약 조건, 관련 데이터, 대상 청중 (예: 최신 금융 감독원 가이드라인 적용)
    - **Output (출력 형식)**: 최종 결과물의 포맷과 톤 (예: Markdown 표 형식, 존댓말)
- **발표자 스크립트 (Presenter Script)**
  - "세계적으로 가장 신뢰받는 프롬프트 구조화 표준은 바로 R-I-C-O 구조입니다. 역할, 지시사항, 배경 맥락, 그리고 출력 형식을 명확히 분리하여 입력할 때, LLM은 지시의 누락 없이 가장 정확하고 일관성 있는 산출물을 만들어 낼 수 있습니다."
- **시각화 가이드 (Visual Guide)**
  - 네 개의 퍼즐 조각이 유기적으로 맞물려 하나의 완벽한 프롬프트 박스를 형성하는 형태의 인포그래픽.

### Slide 22. 도입 질문: 줄글 응답의 한계

  - **문제점 진단**:
    - **가독성 저하**: 긴 텍스트 덩어리는 핵심 파악에 시간이 걸림.
    - **재사용 불가**: 텍스트 사이에 숨어 있는 일정, 담당자 정보를 컴퓨터가 자동으로 추출해 내기 어려움.

  - 하나의 거대한 텍스트 상자(단일 프롬프트)와 세 개의 연결된 작은 박스(Workflow)를 연결 핀 기호와 함께 대조.

### Slide 33. 핵심 설명: 프롬프트 체이닝 (Prompt Chaining)
- **슬라이드 내용 (Slide Content)**
  - **Prompt Chaining의 정의**:
    - 앞 단계의 출력값(Output)이 다음 단계의 입력값(Input)으로 유입되도록 사슬(Chain)처럼 연결한 다단계 AI 처리 방식.
  - **구조도**:
    ```text
    [Input 원본] ──> (Step 1: 요약) ──> [요약본] ──> (Step 2: 추출) ──> [핵심키워드] ──> (Step 3: 작성) ──> [최종 메일]
    ```
  - **장점**: 중간 결과물을 사람이 확인하고 수정(Human-in-the-loop)한 뒤 다음 단계로 보낼 수 있음.
- **발표자 스크립트 (Presenter Script)**
  - "프롬프트 체이닝은 에이전트의 꽃입니다. 전 단계가 끝난 결과 데이터를 다음 단계의 연료로 공급하는 파이프라인을 구축하는 것이죠. 이렇게 하면 중간에 사람이 개입해 잘못된 정보를 올바르게 고쳐 준 후 다음 단계로 진행시킬 수도 있어 안전성이 비약적으로 증가합니다."
- **시각화 가이드 (Visual Guide)**
  - 체인 링크 아이콘이 서로 맞물려 데이터가 흐르는 동적인 라인 애니메이션 형태 시각화.

### Slide 34. 핵심 설명: Workflow 구조의 3요소 (TSF)
- **슬라이드 내용 (Slide Content)**
  - **Workflow 설계 프레임워크**:
    - **1. Task (과제)**: 해결하고자 하는 궁극적인 업무 목표 (예: 고객 피드백 분석 보고서 작성)
    - **2. Step (단계)**: 과제를 달성하기 위해 나눈 개별 처리 단위. 각 Step은 고유의 페르소나와 프롬프트를 가짐 (예: Step 1 감정 분류 -> Step 2 핵심 불만 추출)
    - **3. Flow (흐름)**: Step들이 실행되는 순서와 논리적 연동 방식 (예: 순차 실행, 병렬 처리, 루프)
- **발표자 스크립트 (Presenter Script)**
  - "우리는 워크플로우를 기획할 때 항상 세 가지 레벨로 사고해야 합니다. 전체 목표인 Task, 세부 공정인 Step, 그리고 이들의 결합 질서인 Flow입니다. 이 세 가지가 제대로 매핑되어야 에이전트가 꼬이지 않고 올바르게 목적지에 도달합니다."
- **시각화 가이드 (Visual Guide)**
  - 상위 목표(Task) 아래에 3개의 계층적 박스(Step 1, 2, 3)가 Flow 선으로 정렬된 트리형 다이어그램.

### Slide 43. 핵심 설명: 조건 분기 (Conditional Branching)
- **슬라이드 내용 (Slide Content)**
  - **조건 분기의 개념**:
    - 인풋 데이터의 성격이나 LLM의 판단 결과에 따라 실행 경로(Path)를 다르게 결정하는 논리 구조.
  - **에이전트 내 작동 방식**:
    1. **분류 단계**: LLM이 주어진 기준표에 의거해 텍스트를 특정 카테고리로 매핑.
    2. **라우팅 단계**: 분류 결과값(예: `Negative`, `Inquiry`)에 매칭되는 후속 프롬프트나 시스템 API를 자동 실행.
- **발표자 스크립트 (Presenter Script)**
  - "조건 분기는 에이전트에게 상황 판단의 지도를 쥐어주는 것입니다. 인풋 데이터가 들어오면 LLM이 판사 역할을 수행해 '이것은 항의 메일이다' 판결을 내리고, 시스템은 그 결과 카테고리에 맞는 전용 라인으로 데이터를 자동으로 전달합니다."
- **시각화 가이드 (Visual Guide)**
  - 다이아몬드형 의사결정 아이콘(Decision Tree)을 중심으로 좌측 입력선이 우측 3개의 갈래길로 갈라지는 다이어그램.

### Slide 44. 핵심 설명: 정밀 분류 기준서 설계 (Classification Matrix)
- **슬라이드 내용 (Slide Content)**
  - **분류 기준 4대 영역**:
    - **1. 긍정 (Positive)**: 강의 만족, 시스템 칭찬, 재구매 및 추천 의사 표명.
    - **2. 부정 (Negative)**: 시스템 오류 호소, 불친절 불만, 환불 요구, 강력한 항의.
    - **3. 문의 (Inquiry)**: 일정 확인, 가격 문의, 가이드라인 링크 요청 등 단순 질문.
    - **4. 복합 (Mixed)**: 칭찬과 불만 혹은 질문이 결합된 형태 (예: "강의는 좋은데 결제창이 안 뜸").
- **발표자 스크립트 (Presenter Script)**
  - "분류를 잘 시키기 위해서는 프롬프트 안에 '분류 기준표'를 아주 깐깐하게 정의해야 합니다. 단순히 '좋은 말', '나쁜 말' 수준으로 주면 LLM은 흔들립니다. 어떤 감정과 키워드가 들어와야 부정이고 긍정인지 그 스펙을 정확하게 설명해 주어야 합니다."
- **시각화 가이드 (Visual Guide)**
  - 4대 영역의 분류와 각 영역별 대표 키워드가 정리된 격자형 매트릭스 표.

### Slide 45. 핵심 설명: 예외 처리와 가드레일 (Exception Handling)
- **슬라이드 내용 (Slide Content)**
  - **예외 발생 시나리오와 대응 설계**:
    - **시나리오 A**: 스팸 메일이나 무의미한 텍스트 유입 -> **대응**: `Unclassified`로 처리하고 관리자에게 넘김.
    - **시나리오 B**: 정보 부족 문의 (예: "언제 하나요?") -> **대응**: `"어떤 과정을 의미하시나요?"` 추가 질문 템플릿 트리거.
    - **시나리오 C**: 복합 의도 유입 -> **대응**: 부정 감정을 우선순위로 삼아 긴급 대응 부서로 우선 배정.
- **발표자 스크립트 (Presenter Script)**
  - "현실 세계의 데이터는 지저분합니다. 아무리 분류 기준을 잘 짜도 스팸이나 엉뚱한 입력이 들어오기 마련이죠. 이런 예외 상황에서 시스템이 뻗어버리거나 엉뚱한 소리를 하지 않도록 '예외 대응 예비 경로'를 미리 구축해 두는 것이 프로의 설계입니다."
- **시각화 가이드 (Visual Guide)**
  - '정상 데이터 벨트' 옆에 별도로 분리되는 '예외 처리 보관함(Fallback Box)' 일러스트레이션.

### Slide 46. 중요 기법: 판단 이유(Reasoning) 동시 출력
- **슬라이드 내용 (Slide Content)**
  - **Reasoning 출력의 가치**:
    - AI가 분류 결과를 낼 때 그 근거(이유)를 함께 출력하도록 프롬프트를 설계함.
  - **출력 스키마 예시**:
    ```json
    {
      "classification": "Negative",
      "reason": "고객이 결제 과정 중 '두 번 청구'라는 명확한 금전적 오류를 언급하였고, '확인 요망'이라며 신속한 조치를 촉구하고 있으므로 부정 및 긴급 건으로 분류함."
    }
    ```
    - **이점**: 현업 담당자가 결과의 타당성을 즉시 검토할 수 있으며, 프롬프트 디버깅이 쉬워짐.
- **발표자 스크립트 (Presenter Script)**
  - "AI에게 '그냥 결과만 말해'라고 하면 검증할 방법이 없습니다. 반드시 '왜 그렇게 생각했는지 이유를 적어줘'라고 지시해야 합니다. 이 판단 이유는 나중에 강사나 운영진이 에이전트의 로직을 튜닝할 때 엄청난 힌트가 되며, 현업 실무자에게도 신뢰감을 줍니다."
- **시각화 가이드 (Visual Guide)**
  - JSON 응답 코드 내부에서 `"reason"` 키를 돋보기 기호로 크게 확대해 보여주는 연출.

### Slide 53. 핵심 설명: 자동화 업무 선정의 3대 필터 (R-R-D)
- **슬라이드 내용 (Slide Content)**
  - **적합성 판정 지표**:
    - **1. 반복성 (Repetitive)**: 매주 혹은 매일 발생하여 자동화 시 시간 절감 효과가 큰 작업인가?
    - **2. 규칙성 (Rule-based)**: 처리 로직이나 판단 조건(If-Else)을 인간이 명확한 문장으로 설명할 수 있는가?
    - **3. 데이터성 (Data-driven)**: 일의 입력값과 결과물이 텍스트, 표, JSON 등 디지털 문서 형태로 깨끗이 정의되는가?
- **발표자 스크립트 (Presenter Script)**
  - "우리는 세 가지 필터, 즉 '반복성, 규칙성, 데이터성'을 통과하는 일만 골라내야 합니다. '대표님 마음에 드는 기획안 작성' 같은 일은 규칙성과 데이터성이 모호해 에이전트에 부적합합니다. 반면 '인입된 고객 가입 서류 중 필수 정보 누락 여부 검사'는 세 필터를 완벽히 통과하는 아주 훌륭한 사냥감입니다."
- **시각화 가이드 (Visual Guide)**
  - 3단계 깔때기(Funnel) 모양의 다이어그램. 각 층에 반복성, 규칙성, 데이터성이라는 레이블이 붙어 있고 통과된 업무가 보석 아이콘으로 나오는 비주얼.

### Slide 58. 발표 가이드: 3분 엘리베이터 피칭(Elevator Pitch)
