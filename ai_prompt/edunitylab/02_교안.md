## 3. 1일차 교안

### 1차시. AI · LLM · Agent 개념 이해

#### 도입

- 질문:
  - "ChatGPT에게 질문하는 것과 Agent를 설계하는 것은 무엇이 다를까요?"
  - "AI가 답을 주는 것에서 한 단계 더 나아가 실제 업무를 처리하려면 무엇이 필요할까요?"
- 전달 메시지:
  - LLM은 언어를 생성하고, Agent는 목표를 위해 입력, 판단, 실행, 출력을 연결한다.

#### 핵심 설명

- AI: 사람의 지능적 활동을 기계가 수행하도록 하는 넓은 개념
- LLM: 텍스트를 이해하고 생성하는 언어 모델
- Agent: 목표를 달성하기 위해 입력을 해석하고, 판단하고, 필요한 행동을 수행하는 실행 구조
- Agent 구성:
  - Input: 사용자의 요청 또는 데이터
  - Reasoning: 판단, 추론, 분류, 요약
  - Action: 도구 사용, 저장, 알림, 다음 단계 호출
  - Output: 사용자에게 제공되는 결과 또는 업무 산출물

#### 실습 진행 (40분)

**준비**: 다음 고객 불만 메일 샘플을 제공한다.

```
제목: 배송 지연에 대한 항의

안녕하세요, 지난주 주문한 제품이 아직도 배송되지 않았습니다.
주문 시 2-3일 내 배송이라고 했는데 벌써 일주일이 지났습니다.
연락도 없고, 배송 조회도 안 됩니다. 환불해주세요.
```

**단계 1: 일반 질문 (10분)**

- 프롬프트: "고객 불만 메일에 답장해줘."
- 예상 출력: 일반적인 사과 문장
- **강사 팁**: 결과가 어떤 형태로 나오는지 관찰하게 한다.

**단계 2: 역할 기반 질문 (10분)**

- 프롬프트: "너는 친절한 고객지원 담당자야. 고객 불만 메일에 답장해줘."
- 예상 출력: 고객지원 특유의 친절한 말투와 공감 표현
- **강사 팁**: 말투와 태도의 변화를 비교하게 한다.

**단계 3: 실행 요청형 질문 (15분)**

- 프롬프트: "고객 불만 메일을 분석하고, 불만 원인, 사과 문장, 해결 방안, 후속 조치로 나누어 답장 초안을 작성해줘."
- 예상 출력: 구조화된 분석과 체계적인 답변
- **강사 팁**: 이것이 Agent의 '실행' 개념임을 강조한다.

**단계 4: 비교 및 정리 (5분)**

- 세 결과를 나란히 놓고 차이점을 비교한다.

#### 공유 질문

- 어느 입력이 가장 업무에 바로 쓰기 쉬웠는가?
- 결과 차이를 만든 표현은 무엇이었는가?
- Agent 관점에서 "실행"에 해당하는 부분은 무엇인가?

#### 정리

- 좋은 Agent 설계는 좋은 질문보다 한 단계 더 넓다.
- Agent 설계는 입력, 판단, 실행, 출력의 구조를 만드는 일이다.

### 2차시. 프롬프트 구조화 & Persona 설계

#### 도입

- 질문:
  - "같은 요청을 해도 어떤 역할을 부여하느냐에 따라 결과가 달라지는 이유는 무엇일까요?"
- 전달 메시지:
  - Persona는 응답의 관점, 말투, 판단 기준, 전문성을 바꾸는 장치다.

#### 핵심 설명

- 프롬프트 기본 구조:
  - Role: 어떤 역할로 답할 것인가
  - Instruction: 무엇을 수행할 것인가
  - Context: 어떤 배경과 조건을 고려할 것인가
  - Output: 어떤 형식으로 결과를 낼 것인가
- 좋은 Persona의 조건:
  - 역할이 구체적이다.
  - 판단 기준이 포함되어 있다.
  - 대상 독자와 상황이 명확하다.

#### 실습 진행 (40분)

**준비**: 다음 업무 시나리오를 제공한다.

```
신규 AI 서비스 'Antigravity'가 출시됩니다.
타겟: 중소기업 마케팅 담당자
주요 기능: AI 기반 마케팅 콘텐츠 자동 생성
목적: 서비스 출시 안내문 작성
```

**단계 1: 상담원 Persona (10분)**

- 프롬프트:

```
너는 고객 상담원이야. 친절하고 부드러운 말투로
신규 서비스 출시 안내문을 작성해줘.
```

- 예상 출력: 고객 중심의 친절한 안내, 기능 설명보다 혜택 강조

**단계 2: 기획자 Persona (10분)**

- 프롬프트:

```
너는 서비스 기획자야. 전문적이고 논리적인 말투로
신규 서비스 출시 안내문을 작성해줘.
```

- 예상 출력: 기능 상세 설명, 차별점 강조, 기술적 장점

**단계 3: 전문가 Persona (10분)**

- 프롬프트:

```
너는 AI 마케팅 분야의 전문가야. 신뢰감 있는 말투로
신규 서비스 출시 안내문을 작성해줘.
```

- 예상 출력: 시장 분석, 트렌드 언급, ROI 강조

**단계 4: 최적 Persona 조합 (10분)**

- 각 Persona의 장점을 조합하여 새로운 프롬프트 작성
- 예: "너는 AI 마케팅 전문가이자 고객 중심의 기획자야..."
- **강사 팁**: 역할만 바꾸는 것보다 판단 기준을 포함하는 것이 중요함을 강조한다.

#### 공유 질문

- 어떤 Persona가 가장 유용했는가?
- 역할만 바꿨을 때와 판단 기준까지 넣었을 때 차이가 있었는가?
- 업무에서 자주 쓸 수 있는 Persona는 무엇인가?

#### 정리

- Persona는 단순한 말투 설정이 아니라 결과의 관점을 설계하는 방법이다.
- 역할, 대상, 기준을 함께 넣으면 결과 품질이 안정된다.

### 3차시. 출력 구조 제어 & 데이터화

#### 도입

- 질문:
  - "AI가 좋은 답을 줬더라도 복사해서 업무에 쓰기 어렵다면 무엇이 부족한 걸까요?"
- 전달 메시지:
  - 업무에서 중요한 것은 좋은 문장뿐 아니라 재사용 가능한 구조다.

#### 핵심 설명

- 출력 형식의 종류:
  - 표: 비교, 목록화, 관리에 적합
  - Bullet: 빠른 요약과 공유에 적합
  - JSON: 시스템 연동과 데이터 처리에 적합
- 출력 제어 요소:
  - 항목명
  - 순서
  - 누락 시 처리 방식
  - 예시 형식

#### 실습 진행 (40분)

**준비**: 다음 회의록 샘플을 제공한다.

```
2024년 1월 15일 마케팅 팀 주간 회의

참석자: 김팀장, 이대리, 박사원

1. 지난주 캠페인 성과 분석
   - 클릭률 3.2% 달성 (목표 2.5% 초과)
   - 전환율 1.8% (목표 미달)
   - 개선 필요: 랜딩 페이지 최적화

2. 신규 캠페인 기획
   - 타겟: 2030 여성 직장인
   - 채널: 인스타그램, 블로그
   - 예산: 500만원
   - 담당: 이대리
   - 마감: 1월 25일

3. 다음주 일정
   - 화요일: 크리에이티브 회의
   - 목요일: A/B 테스트 결과 검토
```

**단계 1: 표 형식 변환 (10분)**

- 프롬프트:

```
회의록을 표 형식으로 정리해줘.
컬럼: 구분, 내용, 담당자, 기한
```

- 예상 출력:

```
| 구분 | 내용 | 담당자 | 기한 |
|------|------|--------|------|
| 성과 분석 | 캠페인 성과 분석 | 김팀장 | 완료 |
| 기획 | 신규 캠페인 기획 | 이대리 | 1/25 |
| 일정 | 크리에이티브 회의 | 전체 | 화요일 |
```

- **강사 팁**: 표는 관리와 추적에 적합하다.

**단계 2: Bullet 형식 변환 (10분)**

- 프롬프트:

```
회의록을 불렛 포인트로 요약해줘.
```

- 예상 출력:

```
- 성과 분석: 클릭률 3.2% 달성, 전환율 개선 필요
- 신규 캠페인: 2030 여성 직장인 타겟, 예산 500만원
- 일정: 화요일 크리에이티브 회의, 목요일 A/B 테스트
```

- **강사 팁**: Bullet은 빠른 공유와 요약에 적합하다.

**단계 3: JSON 형식 변환 (10분)**

- 프롬프트:

```
회의록을 JSON 형식으로 변환해줘.
구조: {agenda: [], action_items: [], schedule: []}
```

- 예상 출력:

```json
{
  "agenda": [
    { "type": "성과 분석", "content": "캠페인 성과 분석" },
    { "type": "기획", "content": "신규 캠페인 기획" }
  ],
  "action_items": [{ "task": "랜딩 페이지 최적화", "assignee": "이대리" }],
  "schedule": [{ "date": "화요일", "event": "크리에이티브 회의" }]
}
```

- **강사 팁**: JSON은 시스템 연동과 자동화에 필수적이다.

**단계 4: 활용성 비교 (10분)**

- 각 형식의 장단점과 사용 목적을 논의한다.

#### 공유 질문

- 사람이 읽기 좋은 형식은 무엇인가?
- 시스템에 저장하기 좋은 형식은 무엇인가?
- 업무 목적에 따라 출력 형식을 어떻게 선택해야 하는가?

#### 정리

- 출력 형식은 결과의 활용 방식을 결정한다.
- Agent 설계에서는 결과를 어디에 쓸지 먼저 정해야 한다.

## 4. 2일차 교안

### 4차시. Agent 흐름 설계: Prompt에서 Workflow로

#### 도입

- 질문:
  - "하나의 긴 프롬프트로 처리하는 것과 여러 단계로 나누어 처리하는 것은 어떤 차이가 있을까요?"
- 전달 메시지:
  - 복잡한 문제는 한 번에 해결하기보다 단계별 흐름으로 나누면 안정적이다.

#### 핵심 설명

- Prompt Chaining:
  - 한 단계의 결과가 다음 단계의 입력이 되는 구조
- Task, Step, Flow:
  - Task: 해결할 전체 문제
  - Step: 문제를 나눈 처리 단위
  - Flow: 단계가 연결되는 순서
- 예시 흐름:
  - 회의록 입력
  - 핵심 내용 요약
  - 결정사항 추출
  - 액션 아이템 생성

#### 실습 진행 (40분)

**준비**: 앞서 사용한 회의록 샘플을 활용한다.

**단계 1: 단일 프롬프트 접근 (10분)**

- 프롬프트:

```
이 회의록에서 요약, 핵심 이슈, 실행 항목을 한 번에 추출해줘.
```

- 예상 출력: 모든 내용이 섞인 긴 텍스트
- **강사 팁**: 한 번에 처리할 때의 한계를 관찰하게 한다.

**단계 2: 단계형 접근 - 1단계 요약 (10분)**

- 프롬프트:

```
회의록을 3문장 이내로 요약해줘.
```

- 출력 저장: 다음 단계에서 사용

**단계 3: 단계형 접근 - 2단계 핵심 이슈 (10분)**

- 프롬프트:

```
[1단계 요약 결과]를 바탕으로 핵심 이슈 3가지를 추출해줘.
```

- 출력 저장: 다음 단계에서 사용

**단계 4: 단계형 접근 - 3단계 실행 항목 (10분)**

- 프롬프트:

```
[2단계 핵심 이슈]를 바탕으로 실행 항목을 담당자별로 정리해줘.
```

- **강사 팁**: 각 단계의 출력이 다음 단계의 입력이 되는 구조를 강조한다.

#### 공유 질문

- 단계로 나누었을 때 결과가 더 명확해진 부분은 무엇인가?
- 어느 단계에서 정보 손실이 생겼는가?
- 다음 단계로 넘겨야 하는 중간 결과는 무엇인가?

#### 정리

- Workflow 설계는 일을 잘게 쪼개고 순서를 정하는 과정이다.
- Agent는 각 단계의 목적과 입출력이 명확할수록 안정적으로 동작한다.

### 5차시. 조건 분기 & 판단형 Agent 설계

#### 도입

- 질문:
  - "모든 입력에 같은 답을 주는 Agent는 실무에서 충분할까요?"
- 전달 메시지:
  - 실무 Agent는 입력을 판단하고 상황에 따라 다른 흐름을 선택해야 한다.

#### 핵심 설명

- 조건 분기:
  - 입력값이나 판단 결과에 따라 다른 경로를 선택하는 구조
- 분류 기준:
  - 긍정: 만족, 칭찬, 추천 의사
  - 부정: 불만, 오류, 환불, 항의
  - 문의: 정보 요청, 방법 질문, 확인 요청
- 예외 처리:
  - 애매한 입력
  - 복수 의도가 섞인 입력
  - 정보가 부족한 입력

#### 실습 진행 (40분)

**준비**: 다음 고객 리뷰 샘플 10개를 배포한다.

```
1. "정말 좋아요! 다음에 또 이용할게요." (긍정)
2. "배송이 너무 늦어요. 환불해주세요." (부정)
3. "이 제품 사용법이 어떻게 되나요?" (문의)
4. "가격 대비 퀄리티가 최고예요!" (긍정)
5. "고객센터 연락이 안 돼요. 너무 화나네요." (부정)
6. "재입고什么时候?" (문의 - 애매함)
7. "친구들에게 추천하고 싶어요." (긍정)
8. "하자가 있어요. 교환 가능한가요?" (부정)
9. "회원가입 방법 알려주세요." (문의)
10. "좋아요. 근데 배송만 좀 빨랐으면..." (긍정+부정 혼합)
```

**단계 1: 분류 기준 작성 (10분)**

- 팀별로 긍정/부정/문의 분류 기준 정의
- 예:
  - 긍정: 만족, 칭찬, 추천, 재구매 의사
  - 부정: 불만, 항의, 환불, 교환, 하자
  - 문의: 정보 요청, 사용법, 재입고, 가입 방법

**단계 2: 분류 Agent 설계 (10분)**

- 프롬프트:

```
너는 고객 리뷰 분석 전문가야.
리뷰를 긍정/부정/문의 중 하나로 분류하고,
판단 이유를 함께 설명해줘.

분류 기준:
- 긍정: 만족, 칭찬, 추천 의사
- 부정: 불만, 오류, 환불, 항의
- 문의: 정보 요청, 방법 질문, 확인 요청

출력 형식:
리뷰: [원문]
분류: [긍정/부정/문의]
이유: [판단 근거]
```

**단계 3: 분류 실행 및 애매한 케이스 실험 (15분)**

- 10개 리뷰를 분류
- 애매한 케이스(6번, 10번)의 결과를 확인
- 기준을 수정하여 재실험

**단계 4: 팀별 비교 (5분)**

- 각 팀의 정확도와 기준 차이 공유
- **강사 팁**: 기준의 명확성이 결과의 일관성을 결정함을 강조한다.

#### 공유 질문

- 분류 기준이 모호했던 사례는 무엇인가?
- 판단 이유를 함께 출력하면 어떤 점이 좋아지는가?
- 예외 입력은 어떻게 처리하는 것이 좋은가?

#### 정리

- 조건 분기는 Agent의 판단력을 설계하는 부분이다.
- 기준과 예외를 명확히 할수록 결과가 안정된다.

### 6차시. 업무 자동화 Agent 설계: Mini Project

#### 도입

- 질문:
  - "여러분의 업무 중 AI Agent에게 맡기고 싶은 반복 업무는 무엇인가요?"
- 전달 메시지:
  - 자동화할 업무는 멋있어 보이는 업무보다 반복성, 규칙성, 데이터성이 있는 업무가 적합하다.

#### 핵심 설명

- 자동화 대상 선정 기준:
  - 반복성: 자주 발생하는가
  - 규칙성: 판단 기준이나 처리 방식이 설명 가능한가
  - 데이터성: 입력과 출력이 문서, 텍스트, 표 등으로 정리 가능한가
- Agent 설계서 구성:
  - 문제 정의
  - 사용자 또는 대상 업무
  - 입력 데이터
  - 처리 단계
  - 조건 분기
  - 출력 결과
  - 기대 효과

#### 실습 진행 (60분)

**단계 1: 반복 업무 후보 도출 (15분)**

- 개인 또는 팀별로 자신의 업무 중 반복 업무 3개 작성
- 예시:
  - 고객 문의 이메일 분류 및 답변 초안 작성
  - 주간 보고서 작성을 위한 데이터 수집 및 정리
  - SNS 게시물 댓글 분석 및 감성 분류
  - 제안서 검토 요약 및 평가 점수 산출
  - 회의록 정리 및 액션 아이템 추출

**단계 2: 자동화 적합성 평가 (10분)**

- 다음 기준으로 평가하여 1개 선정:
  - 반복성: 일주일에 몇 번 발생하는가?
  - 규칙성: 판단 기준을 설명할 수 있는가?
  - 데이터성: 입력/출력이 텍스트로 정리 가능한가?
  - 시간 절감: 자동화 시 얼마나 시간을 아낄 수 있는가?

**단계 3: Agent 설계서 작성 (25분)**

- 다음 구조로 설계서 작성:

```
[Agent 설계서]

1. 문제 정의
   - 현재 업무의 문제점
   - 자동화로 해결하고자 하는 목표

2. 사용자 및 대상 업무
   - 주요 사용자
   - 업무 빈도

3. 입력 데이터
   - 입력 형태 (이메일, 보고서, 댓글 등)
   - 입력 예시
   - 입력 빈도

4. 처리 단계 (Workflow)
   - Step 1: [목적] - [상세 설명]
   - Step 2: [목적] - [상세 설명]
   - Step 3: [목적] - [상세 설명]

5. 조건 분기
   - 분류 기준
   - 각 분기별 처리 방식
   - 예외 처리 방안

6. Persona 설계
   - 역할
   - 판단 기준
   - 말투 및 태도

7. 출력 형식
   - 출력 구조 (표, JSON, Bullet 등)
   - 활용 목적

8. 기대 효과
   - 시간 절감 예상
   - 품질 개선 기대
   - 확장 가능성
```

**단계 4: 발표 및 피드백 (10분)**

- 팀별 3분 발표
- 피드백 질문:
  - 입력이 충분히 명확한가?
  - 처리 단계가 너무 복잡하지 않은가?
  - 조건 분기가 실제 업무를 반영하는가?
  - 출력이 실제로 활용 가능한가?

#### 공유 질문

- 이 업무는 왜 자동화에 적합한가?
- 입력이 명확한가?
- 처리 단계가 너무 넓거나 모호하지 않은가?
- 결과를 어디에 활용할 수 있는가?

#### 정리

- Mini Project의 핵심은 완성된 구현보다 설계의 명확성이다.
- 좋은 Agent 설계서는 문제, 입력, 처리, 출력, 기대 효과가 한눈에 보여야 한다.

## 5. 마무리 멘트 예시

오늘과 내일의 목표는 AI를 잘 쓰는 사람이 되는 것을 넘어, AI가 일하는 방식을 설계하는 사람이 되는 것입니다. 좋은 Agent는 한 번에 만들어지지 않습니다. 질문을 바꾸고, 역할을 바꾸고, 출력 형식을 바꾸고, 흐름을 나누고, 조건을 추가하면서 점점 업무에 가까워집니다. Mini Project에서는 그 과정을 실제 업무 문제에 적용해보겠습니다.

## 6. 강사를 위한 추가 팁

### 시간 관리

- 실습 시간은 엄수하되, 학습자 속도에 따라 유연하게 조절
- 공유 시간에 너무 많은 팀이 발표하지 않도록 3-4팀으로 제한
- 마지막 10분은 여유를 두고 정리

### 질문 처리

- 기술적 질문보다 개념적 질문에 집중
- "어떻게"보다 "왜"에 대한 답변 우선
- 모르는 질문은 "나중에 알아보겠다"고 솔직하게 응답

### 실습 지원

- 프롬프트 작성 시 직접 타이핑하게 하고, 필요한 경우에만 수정 제안
- 결과가 좋지 않을 때 바로 답을 주지 말고, 어디를 수정할지 질문
- 팀별로 순회하며 막히는 부분 지원

### 피드백 방법

- "좋습니다/나쁩니다"보다 구체적인 관찰 제공
- 다른 팀의 결과와 비교하여 인사이트 제공
- 실패 사례도 학습 기회로 활용

### 기술적 문제 대비

- 인터넷 연결 불안정 대비 오프라인 자료 준비
- Antigravity 장애 시 대체 도구 안내
- 노트북 충전기 미携带 대비 여분 콘센트 확인
