# 에듀니티랩 1,2일차 강사용 교안

## 1. 강의 운영 원칙

- 강의 톤: 개념은 짧게, 실습은 구체적으로, 공유는 빠르게 진행한다.
- 핵심 메시지: Agent는 답변을 잘하는 도구가 아니라 입력을 받아 판단하고 실행 결과를 만드는 시스템이다.
- 진행 방식: 설명 20%, 실습 60%, 공유와 피드백 20%를 기본 비율로 한다.
- 산출물 연결: 각 차시 결과물을 6차시 Mini Project 설계서의 일부로 재사용한다.

## 2. 준비물

- 강사용:
  - Antigravity 접속 및 시연 계정
  - 예시 프롬프트와 예시 응답
  - 회의록 샘플 텍스트
  - 고객 리뷰 샘플 10개
  - Agent 설계서 템플릿
- 학습자:
  - 노트북
  - 인터넷 연결
  - Antigravity 활용 환경
  - 개인 또는 팀별 실습 기록 문서

## 3. 1일차 교안

### 1차시. AI · LLM · Agent 개념 이해

#### 도입

- 질문:
  - "ChatGPT에게 질문하는 것과 Agent를 설계하는 것은 무엇이 다를까요?"
  - "AI가 답을 주는 것에서 한 단계 더 나아가 실제 업무를 처리하려면 무엇이 필요할까요?"
- 전달 메시지:
  - LLM은 언어를 생성하고, Agent는 목표를 위해 입력, 판단, 실행, 출력을 연결한다.

#### 핵심 설명

- AI: 사람의 지능적 활동을 기계가 수행하도록 하는 넓은 개념
- LLM: 텍스트를 이해하고 생성하는 언어 모델
- Agent: 목표를 달성하기 위해 입력을 해석하고, 판단하고, 필요한 행동을 수행하는 실행 구조
- Agent 구성:
  - Input: 사용자의 요청 또는 데이터
  - Reasoning: 판단, 추론, 분류, 요약
  - Action: 도구 사용, 저장, 알림, 다음 단계 호출
  - Output: 사용자에게 제공되는 결과 또는 업무 산출물

#### 실습 진행

1. 일반 질문을 입력한다.
   - 예: "고객 불만 메일에 답장해줘."
2. 역할 기반 질문을 입력한다.
   - 예: "너는 친절한 고객지원 담당자야. 고객 불만 메일에 답장해줘."
3. 실행 요청형 질문을 입력한다.
   - 예: "고객 불만 메일을 분석하고, 불만 원인, 사과 문장, 해결 방안, 후속 조치로 나누어 답장 초안을 작성해줘."
4. 세 결과를 비교한다.

#### 공유 질문

- 어느 입력이 가장 업무에 바로 쓰기 쉬웠는가?
- 결과 차이를 만든 표현은 무엇이었는가?
- Agent 관점에서 "실행"에 해당하는 부분은 무엇인가?

#### 정리

- 좋은 Agent 설계는 좋은 질문보다 한 단계 더 넓다.
- Agent 설계는 입력, 판단, 실행, 출력의 구조를 만드는 일이다.

### 2차시. 프롬프트 구조화 & Persona 설계

#### 도입

- 질문:
  - "같은 요청을 해도 어떤 역할을 부여하느냐에 따라 결과가 달라지는 이유는 무엇일까요?"
- 전달 메시지:
  - Persona는 응답의 관점, 말투, 판단 기준, 전문성을 바꾸는 장치다.

#### 핵심 설명

- 프롬프트 기본 구조:
  - Role: 어떤 역할로 답할 것인가
  - Instruction: 무엇을 수행할 것인가
  - Context: 어떤 배경과 조건을 고려할 것인가
  - Output: 어떤 형식으로 결과를 낼 것인가
- 좋은 Persona의 조건:
  - 역할이 구체적이다.
  - 판단 기준이 포함되어 있다.
  - 대상 독자와 상황이 명확하다.

#### 실습 진행

1. 하나의 업무 질문을 정한다.
   - 예: "신규 서비스 출시 안내문을 작성해줘."
2. 상담원 Persona로 실행한다.
3. 기획자 Persona로 실행한다.
4. 전문가 Persona로 실행한다.
5. 결과를 비교하고 최적 Persona를 조합한다.

#### 공유 질문

- 어떤 Persona가 가장 유용했는가?
- 역할만 바꿨을 때와 판단 기준까지 넣었을 때 차이가 있었는가?
- 업무에서 자주 쓸 수 있는 Persona는 무엇인가?

#### 정리

- Persona는 단순한 말투 설정이 아니라 결과의 관점을 설계하는 방법이다.
- 역할, 대상, 기준을 함께 넣으면 결과 품질이 안정된다.

### 3차시. 출력 구조 제어 & 데이터화

#### 도입

- 질문:
  - "AI가 좋은 답을 줬더라도 복사해서 업무에 쓰기 어렵다면 무엇이 부족한 걸까요?"
- 전달 메시지:
  - 업무에서 중요한 것은 좋은 문장뿐 아니라 재사용 가능한 구조다.

#### 핵심 설명

- 출력 형식의 종류:
  - 표: 비교, 목록화, 관리에 적합
  - Bullet: 빠른 요약과 공유에 적합
  - JSON: 시스템 연동과 데이터 처리에 적합
- 출력 제어 요소:
  - 항목명
  - 순서
  - 누락 시 처리 방식
  - 예시 형식

#### 실습 진행

1. 회의록 샘플을 제공한다.
2. 표 형식으로 변환한다.
   - 컬럼 예시: 구분, 내용, 담당자, 기한
3. Bullet 형식으로 변환한다.
4. JSON 형식으로 변환한다.
5. 각 형식의 활용성을 비교한다.

#### 공유 질문

- 사람이 읽기 좋은 형식은 무엇인가?
- 시스템에 저장하기 좋은 형식은 무엇인가?
- 업무 목적에 따라 출력 형식을 어떻게 선택해야 하는가?

#### 정리

- 출력 형식은 결과의 활용 방식을 결정한다.
- Agent 설계에서는 결과를 어디에 쓸지 먼저 정해야 한다.

## 4. 2일차 교안

### 4차시. Agent 흐름 설계: Prompt에서 Workflow로

#### 도입

- 질문:
  - "하나의 긴 프롬프트로 처리하는 것과 여러 단계로 나누어 처리하는 것은 어떤 차이가 있을까요?"
- 전달 메시지:
  - 복잡한 문제는 한 번에 해결하기보다 단계별 흐름으로 나누면 안정적이다.

#### 핵심 설명

- Prompt Chaining:
  - 한 단계의 결과가 다음 단계의 입력이 되는 구조
- Task, Step, Flow:
  - Task: 해결할 전체 문제
  - Step: 문제를 나눈 처리 단위
  - Flow: 단계가 연결되는 순서
- 예시 흐름:
  - 회의록 입력
  - 핵심 내용 요약
  - 결정사항 추출
  - 액션 아이템 생성

#### 실습 진행

1. 하나의 업무 텍스트를 준비한다.
2. 1단계에서 요약한다.
3. 2단계에서 핵심 이슈를 추출한다.
4. 3단계에서 실행 항목을 도출한다.
5. 단일 프롬프트 결과와 단계형 결과를 비교한다.

#### 공유 질문

- 단계로 나누었을 때 결과가 더 명확해진 부분은 무엇인가?
- 어느 단계에서 정보 손실이 생겼는가?
- 다음 단계로 넘겨야 하는 중간 결과는 무엇인가?

#### 정리

- Workflow 설계는 일을 잘게 쪼개고 순서를 정하는 과정이다.
- Agent는 각 단계의 목적과 입출력이 명확할수록 안정적으로 동작한다.

### 5차시. 조건 분기 & 판단형 Agent 설계

#### 도입

- 질문:
  - "모든 입력에 같은 답을 주는 Agent는 실무에서 충분할까요?"
- 전달 메시지:
  - 실무 Agent는 입력을 판단하고 상황에 따라 다른 흐름을 선택해야 한다.

#### 핵심 설명

- 조건 분기:
  - 입력값이나 판단 결과에 따라 다른 경로를 선택하는 구조
- 분류 기준:
  - 긍정: 만족, 칭찬, 추천 의사
  - 부정: 불만, 오류, 환불, 항의
  - 문의: 정보 요청, 방법 질문, 확인 요청
- 예외 처리:
  - 애매한 입력
  - 복수 의도가 섞인 입력
  - 정보가 부족한 입력

#### 실습 진행

1. 고객 리뷰 샘플을 배포한다.
2. 긍정, 부정, 문의 분류 기준을 작성한다.
3. 리뷰별 분류 결과와 판단 이유를 출력하게 한다.
4. 애매한 리뷰를 추가해 예외 처리를 실험한다.
5. 팀별 정확도와 기준 차이를 비교한다.

#### 공유 질문

- 분류 기준이 모호했던 사례는 무엇인가?
- 판단 이유를 함께 출력하면 어떤 점이 좋아지는가?
- 예외 입력은 어떻게 처리하는 것이 좋은가?

#### 정리

- 조건 분기는 Agent의 판단력을 설계하는 부분이다.
- 기준과 예외를 명확히 할수록 결과가 안정된다.

### 6차시. 업무 자동화 Agent 설계: Mini Project

#### 도입

- 질문:
  - "여러분의 업무 중 AI Agent에게 맡기고 싶은 반복 업무는 무엇인가요?"
- 전달 메시지:
  - 자동화할 업무는 멋있어 보이는 업무보다 반복성, 규칙성, 데이터성이 있는 업무가 적합하다.

#### 핵심 설명

- 자동화 대상 선정 기준:
  - 반복성: 자주 발생하는가
  - 규칙성: 판단 기준이나 처리 방식이 설명 가능한가
  - 데이터성: 입력과 출력이 문서, 텍스트, 표 등으로 정리 가능한가
- Agent 설계서 구성:
  - 문제 정의
  - 사용자 또는 대상 업무
  - 입력 데이터
  - 처리 단계
  - 조건 분기
  - 출력 결과
  - 기대 효과

#### 실습 진행

1. 개인 또는 팀별로 반복 업무 후보를 3개 작성한다.
2. 자동화 적합성 기준으로 1개를 선정한다.
3. 입력, 처리, 출력 구조를 작성한다.
4. 필요한 Persona, 출력 형식, 조건 분기, Workflow를 설계한다.
5. 팀별로 3분 발표하고 피드백을 받는다.

#### 공유 질문

- 이 업무는 왜 자동화에 적합한가?
- 입력이 명확한가?
- 처리 단계가 너무 넓거나 모호하지 않은가?
- 결과를 어디에 활용할 수 있는가?

#### 정리

- Mini Project의 핵심은 완성된 구현보다 설계의 명확성이다.
- 좋은 Agent 설계서는 문제, 입력, 처리, 출력, 기대 효과가 한눈에 보여야 한다.

## 5. 마무리 멘트 예시

오늘과 내일의 목표는 AI를 잘 쓰는 사람이 되는 것을 넘어, AI가 일하는 방식을 설계하는 사람이 되는 것입니다. 좋은 Agent는 한 번에 만들어지지 않습니다. 질문을 바꾸고, 역할을 바꾸고, 출력 형식을 바꾸고, 흐름을 나누고, 조건을 추가하면서 점점 업무에 가까워집니다. Mini Project에서는 그 과정을 실제 업무 문제에 적용해보겠습니다.
