# 에듀니티랩 1,2일차 실습과제

## 1. 제출물 목록

학습자는 1,2일차 종료 시 다음 산출물을 제출한다.

| 번호 | 제출물 | 형식 | 관련 차시 |
|---:|---|---|---|
| 1 | 질문 방식별 응답 비교표 | 표 또는 문서 | 1차시 |
| 2 | Persona별 프롬프트와 응답 비교 | 표 또는 문서 | 2차시 |
| 3 | 회의록 구조화 결과 | 표, Bullet, JSON | 3차시 |
| 4 | 3단계 Workflow 설계표 | 표 또는 흐름도 | 4차시 |
| 5 | 리뷰 분류 Agent 설계안 | 표 또는 문서 | 5차시 |

## 2. 실습과제 1: 질문 방식별 응답 비교

### 목적

AI, LLM, Agent의 차이를 응답 결과 비교를 통해 이해한다.

### 과제 안내

동일한 업무 요청을 세 가지 방식으로 입력하고 결과를 비교한다.

### 입력 예시

```text
고객 불만 메일에 답장해줘.
```

### 수행 단계

1. 일반 질문으로 입력한다.
2. 역할을 부여해 입력한다.
3. 실행 단계와 출력 형식을 포함해 입력한다.
4. 세 결과를 비교한다.
5. Agent 관점에서 가장 좋은 프롬프트로 다시 작성한다.

### 제출 양식

| 구분 | 입력한 프롬프트 | 응답 특징 | 장점 | 한계 |
|---|---|---|---|---|
| 일반 질문 |  |  |  |  |
| 역할 기반 |  |  |  |  |
| 실행 요청형 |  |  |  |  |

### 성찰 질문

- 어떤 프롬프트가 가장 업무에 바로 쓰기 쉬웠는가?
- 결과 차이를 만든 핵심 표현은 무엇인가?
- Agent 설계 관점에서 추가해야 할 요소는 무엇인가?

## 3. 실습과제 2: Persona별 프롬프트 설계

### 목적

Persona가 응답의 관점, 말투, 판단 기준에 미치는 영향을 이해한다.

### 과제 안내

동일한 요청을 세 가지 Persona로 실행하고 결과를 비교한다.

### 입력 예시

```text
신규 서비스 출시 안내문을 작성해줘.
```

### Persona 예시

- 친절한 고객 상담원
- 논리적인 서비스 기획자
- 업계 전문가

### 수행 단계

1. Role, Instruction, Context, Output 구조로 프롬프트를 작성한다.
2. 세 가지 Persona로 각각 실행한다.
3. 응답 결과를 비교한다.
4. 가장 효과적인 Persona를 선택하고 이유를 적는다.
5. 최종 개선 프롬프트를 작성한다.

### 제출 양식

|---|---|---|---|

## 4. 실습과제 3: 회의록 출력 구조화

### 목적

비정형 텍스트를 업무에 활용 가능한 구조화 데이터로 변환한다.

### 회의록 샘플

```text
5월 정기 회의에서 신규 교육 프로그램 홍보 일정이 논의되었다.
홍보 문안은 김민지가 이번 주 금요일까지 초안을 작성하기로 했다.
랜딩 페이지 수정은 박준호가 다음 주 화요일까지 완료한다.
예산은 기존 안에서 10% 증액하는 방향으로 검토하기로 했다.
수강생 모집 공지는 다음 주 수요일 오전에 발송한다.
추가로 교육 만족도 조사 문항을 이서연이 정리하기로 했다.
```

### 수행 단계

1. 회의록을 표 형식으로 변환한다.
2. 회의록을 Bullet 형식으로 요약한다.
3. 회의록을 JSON 형식으로 변환한다.
4. 세 형식의 장단점을 비교한다.

### 표 출력 양식

| 구분 | 내용 | 담당자 | 기한 |
|---|---|---|---|
| 결정사항 |  |  |  |
| 할 일 |  |  |  |

### JSON 출력 예시

```json
{
  "decisions": [],
  "tasks": [
    {
      "task": "",
```

### 성찰 질문

- 시스템에 저장하기 좋은 형식은 무엇인가?
- 출력 형식을 지정할 때 빠뜨리기 쉬운 조건은 무엇인가?

## 5. 실습과제 4: 3단계 Workflow 설계

### 목적

단일 프롬프트를 단계형 Workflow로 확장한다.

### 과제 안내

업무 텍스트를 입력받아 다음 3단계로 처리하는 Agent 흐름을 설계한다.

- 1단계: 요약
- 2단계: 핵심 이슈 추출
- 3단계: 액션 아이템 도출

### 수행 단계

1. 처리할 업무 텍스트를 선택한다.
2. 단계별 목적을 작성한다.
3. 단계별 입력과 출력을 정의한다.
4. 단계별 프롬프트를 작성한다.
5. 최종 결과가 단일 프롬프트보다 나아진 점을 정리한다.

### 제출 양식

| 단계 | 목적 | 입력 | 처리 지시 | 출력 형식 |
|---:|---|---|---|---|
| 1 | 요약 | 원문 |  |  |
| 2 | 핵심 이슈 추출 | 1단계 결과 |  |  |
| 3 | 액션 아이템 도출 | 2단계 결과 |  |  |

### 점검 기준

- 각 단계의 입력과 출력이 명확한가?
- 다음 단계로 넘길 정보가 충분한가?
- 단계가 너무 많거나 불필요하게 나뉘어 있지 않은가?

## 6. 실습과제 5: 리뷰 분류 Agent 설계

### 목적

조건 분기와 판단 기준을 활용해 분류형 Agent를 설계한다.

### 리뷰 샘플

| 번호 | 리뷰 |
|---:|---|
| 1 | 강의 내용이 실무에 바로 적용할 수 있어서 좋았습니다. |
| 2 | 신청 페이지 오류 때문에 결제가 두 번 시도됐습니다. 확인 부탁드립니다. |
| 3 | 다음 교육 일정은 언제 열리나요? |
| 4 | 강사님 설명은 좋았지만 실습 시간이 조금 부족했습니다. |
| 5 | 자료 다운로드 링크가 열리지 않습니다. |

### 분류 기준

- 긍정: 만족, 칭찬, 추천, 재참여 의사
- 부정: 불만, 오류, 불편, 환불, 항의
- 문의: 일정, 방법, 가격, 절차 등 정보 요청
- 복합: 긍정과 부정 또는 문의가 함께 포함된 경우

### 수행 단계

1. 분류 기준을 프롬프트에 명시한다.
2. 리뷰별 분류 결과를 출력한다.
3. 각 분류의 판단 이유를 함께 출력한다.
4. 복합 의도 또는 애매한 입력의 처리 방식을 추가한다.
5. 팀별 결과를 비교한다.

### 제출 양식

| 번호 | 리뷰 요약 | 분류 | 판단 이유 | 후속 조치 |
|---:|---|---|---|---|
| 1 |  |  |  |  |
| 2 |  |  |  |  |
| 3 |  |  |  |  |
| 4 |  |  |  |  |
| 5 |  |  |  |  |

### 예외 처리 규칙

```text
정보가 부족하면 "추가 확인 필요"로 표시한다.
긍정과 부정이 함께 있으면 "복합"으로 분류하고 주요 불편 요소를 우선 기록한다.
문의와 불만이 함께 있으면 "부정"으로 분류하되 후속 조치에 안내 필요 항목을 기록한다.
```

## 8. 평가 루브릭

| 평가 항목 | 우수 | 보통 | 보완 필요 |
|---|---|---|---|
| 문제 정의 | 해결 대상이 구체적이고 업무 맥락이 명확함 | 문제는 있으나 범위가 다소 넓음 | 문제가 모호하거나 Agent로 해결하기 어려움 |
| 입력 설계 | 입력 데이터가 명확하고 예시가 있음 | 입력은 있으나 형식이 불명확함 | 입력이 추상적임 |
| 처리 흐름 | 단계별 목적과 출력이 명확함 | 단계는 있으나 연결이 약함 | 단일 요청 수준에 머무름 |
| 조건 분기 | 판단 기준과 예외 처리가 있음 | 기본 분류 기준만 있음 | 조건이 없거나 모호함 |
| 출력 설계 | 활용 목적에 맞는 출력 형식이 있음 | 출력 형식은 있으나 활용성이 낮음 | 출력이 자유 서술에 가까움 |
| 기대 효과 | 시간 절감, 품질 향상 등 효과가 구체적임 | 기대 효과가 일반적임 | 효과가 불명확함 |

## 9. 제출 전 체크리스트

- AI, LLM, Agent 차이를 내 말로 설명할 수 있다.

- Workflow의 각 단계별 입력과 출력이 명확하다.
- 조건 분기의 판단 기준과 예외 처리를 작성했다.
- Mini Project 설계서에 문제, 입력, 처리, 출력, 기대 효과가 모두 포함되어 있다.
